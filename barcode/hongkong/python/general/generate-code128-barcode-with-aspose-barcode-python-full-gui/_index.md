# 使用 Aspose.Barcode Python 產生 Code128 條碼 – 完整指南

是否曾在 Python 專案中需要 **產生 Code128 條碼**，卻不確定如何嵌入非 ASCII 字元？你並不孤單。許多開發者在嘗試將數字資料與 Unicode 文字（例如俄文或中文）混合於同一條碼時，常會卡關。好消息是：Aspose.Barcode for Python 讓這件事變得輕鬆，且只要正確使用 *barcode with ECI*，即可讓全球掃描器皆能正確辨識。

在本教學中，我們將一步步示範完整、可執行的範例，說明如何使用 Aspose.Barcode Python 函式庫 **產生 Code128 條碼**、如何打造包含 ECI 區段的延伸編碼文字，以及每一行程式碼的意義。完成後，你將擁有可直接執行的腳本、對底層概念的清晰理解，並掌握避免常見陷阱的幾個專業技巧。

## 你將學到

- 在環境中設定 **Aspose.Barcode Python**。  
- 建構混合純數字與 UTF‑8 文字的延伸編碼文字字串（透過 ECI 區段）。  
- 設定 `BarCodeGenerator` 以輸出 **Code128 條碼**。  
- 儲存影像並驗證結果。  
- 處理不同字元集與條碼符號的邊緣情況。

不需要額外文件——所有資訊皆在此。

## 前置條件

在開始之前，請確保你已具備：

1. 已安裝 **Python 3.8+**（函式庫支援 3.6 以上版本）。  
2. 有效的 **Aspose.Barcode for Python via .NET** 授權，或接受評估版的浮水印。  
3. 可使用終端機或 IDE 安裝套件（`pip`）。  
4. 一個用來存放產生 PNG 的資料夾（在程式碼中替換 `YOUR_DIRECTORY`）。

若上述任一項目不熟悉，別擔心——安裝套件只需一條指令，其他皆為純 Python。

## 步驟 1：安裝 Aspose.Barcode for Python

在終端機執行：

```bash
pip install aspose-barcode
```

> **專業提示：** 使用虛擬環境（`python -m venv venv`）可讓相依套件彼此隔離，亦能避免與其他專案的版本衝突。

此套件會在背後載入 .NET 執行環境，首次匯入可能會稍稍延遲。

---

## 步驟 2：建立延伸編碼文字建構器

支援 **barcode with ECI** 的核心在 `ExtCodetextBuilder`。它允許你將純文字段與 ECI 編碼段串接，確保掃描器正確解讀 Unicode 文字。

```python
import aspose.barcode as barcode

# Initialize the builder – it starts empty
ext_builder = barcode.generation.ExtCodetextBuilder()
ext_builder.clear()  # Explicitly clear any residual data (good hygiene)
```

**為何要 clear？**  
若在迴圈中重複使用建構器，殘留的資料會汙染下一個條碼。呼叫 `clear()` 可保證每次都是全新狀態。

---

## 步驟 3：加入純數字段

大多數 Code128 條碼會以數字識別碼開頭——例如訂單號或產品 ID。將其作為 *plain* 段加入，可保持編碼簡潔且緊湊。

```python
# Plain numeric part – 10 digits in this example
ext_builder.add_plain_codetext("1234567890")
```

此方法會自動挑選最適合的 Code128 子集（A、B 或 C）來編碼數字，產生最小的條碼。

---

## 步驟 4：插入含俄文的 ECI 段

接下來的挑戰是：嵌入俄文字符（`Пример текста`）。若未加入 ECI 標記，多數掃描器會錯誤解讀這些位元，導致亂碼。

```python
# ECI value 26 corresponds to UTF‑8 per ISO/IEC 18004
ext_builder.add_eci_codetext(26, "Пример текста")  # UTF‑8 encoded text
```

**發生了什麼？**  
- `26` 告訴條碼後續的位元使用 UTF‑8 編碼。  
- 函式庫會自行處理轉換，你只需傳入 Unicode 字串。  
- 掃描器讀取條碼時，會看到 ECI，切換至 UTF‑8，從而正確顯示俄文。

> **常見問題：** *如果我要使用其他語言怎麼辦？*  
> 使用相對應的 ECI 值（例如 27 代表 ISO‑8859‑1，28 代表 ISO‑8859‑2）。Aspose 文件中有快速查表，使用方式皆相同。

---

## 步驟 5：取得完整的延伸編碼文字

兩個段落加入後，我們取得合併後的字串，這就是要指派給產生器 `code_text` 的值。

```python
# Get the ready‑to‑use extended codetext
extended_code_text = ext_builder.get_extended_codetext()
print("Extended codetext:", extended_code_text)
```

印出的結果大致如下：

```
Extended codetext: 1234567890[ECI26]Пример текста
```

`[ECI26]` 這個標記在最終影像中是不可見的，但對正確解碼至關重要。

---

## 步驟 6：設定 Code128 的 BarCodeGenerator

現在把所有元件串起來。`BarCodeGenerator` 是負責繪製影像的核心。

```python
# Initialize generator
generator = barcode.BarCodeGenerator()
generator.symbology = barcode.Symbology.CODE128   # Primary focus: generate Code128 barcode
generator.code_text = extended_code_text          # Feed the extended codetext

# Optional: tweak appearance
generator.parameters.barcode.x_dimension = 0.5   # Narrower bars for higher density
generator.parameters.barcode.bar_height = 30     # Height in points
```

**為何要設定 `x_dimension`？**  
較小的模組寬度會產生更緊湊的條碼——在空間受限時特別有用。可依印表機 DPI 調整此參數。

---

## 步驟 7：儲存條碼影像

最後，將 PNG 寫入磁碟。改變副檔名即可輸出 JPEG、BMP，甚至 SVG。

```python
output_path = "YOUR_DIRECTORY/extended_code.png"
generator.save(output_path)

print(f"Barcode saved to {output_path}")
```

使用任意檢視器開啟影像，你應該會看到一條乾淨的 Code128 條碼，使用現代讀取器掃描時會回傳 `1234567890Пример текста`。

## 完整腳本 – 複製並執行

以下為完整、獨立的腳本。將其貼到名為 `generate_code128.py` 的檔案，調整 `YOUR_DIRECTORY` 後執行 `python generate_code128.py`。

```python
import aspose.barcode as barcode
import os

# ------------------------------------------------------------------
# Step 1: Prepare output directory (creates it if missing)
# ------------------------------------------------------------------
output_dir = "YOUR_DIRECTORY"
os.makedirs(output_dir, exist_ok=True)

# ------------------------------------------------------------------
# Step 2: Build extended codetext (plain + ECI segment)
# ------------------------------------------------------------------
ext_builder = barcode.generation.ExtCodetextBuilder()
ext_builder.clear()
ext_builder.add_plain_codetext("1234567890")
ext_builder.add_eci_codetext(26, "Пример текста")  # 26 = UTF‑8

extended_code_text = ext_builder.get_extended_codetext()
print("Extended codetext:", extended_code_text)

# ------------------------------------------------------------------
# Step 3: Configure generator for Code128
# ------------------------------------------------------------------
generator = barcode.BarCodeGenerator()
generator.symbology = barcode.Symbology.CODE128
generator.code_text = extended_code_text

# Optional appearance tweaks
generator.parameters.barcode.x_dimension = 0.5
generator.parameters.barcode.bar_height = 30

# ------------------------------------------------------------------
# Step 4: Save the barcode image
# ------------------------------------------------------------------
output_path = os.path.join(output_dir, "extended_code.png")
generator.save(output_path)

print(f"Barcode saved to {output_path}")
```

**預期輸出**（主控台）：

```
Extended codetext: 1234567890[ECI26]Пример текста
Barcode saved to YOUR_DIRECTORY/extended_code.png
```

開啟 `extended_code.png`，即可看到已備妥投入生產的清晰 Code128 條碼。

## 深入探討：ECI 機制如何運作

ECI（Extended Channel Interpretation）是 QR 與條碼標準的一部份，用於在資料流中切換字元編碼。於 Code128 的情境下：

1. 條碼預設使用 ISO‑8859‑1 模式。  
2. 掃描器遇到 ECI 控制碼（本例中的值 26）時，切換至 UTF‑8。  
3. 之後的所有位元皆以 UTF‑8 方式解讀，直到出現另一個 ECI 或資料結束。

因為我們明確呼叫 `add_eci_codetext(26, ...)`，Aspose 會在背後插入正確的控制字元。若省略 ECI，俄文文字會以預設代碼頁編碼，導致大多數掃描器顯示為不可讀的亂碼。

---

## 處理邊緣案例

### 1. 多個 ECI 段落

若需同時嵌入日文與阿拉伯文，可多次呼叫 `add_eci_codetext`，傳入不同的 ECI 值。建構器會依加入順序串接。

```python
ext_builder.add_eci_codetext(27, "Texto en Español")   # ISO‑8859‑1
ext_builder.add_eci_codetext(28, "文字列日本語")       # ISO‑8859‑2 (example)
```

### 2. 長度限制

Code128 最多可編碼 2 023 個字元，但實務上受掃描器記憶體限制，建議保持在 1 000 字元以下，特別是混合 ECI 段時。超過上限會拋出 `ArgumentException`。

### 3. 與舊版掃描器的相容性

部分舊版掃描器會忽略 ECI 控制碼。在此情況下，建議產生兩條條碼——一條僅含數字，另一條僅含 Unicode 文字，或改用支援度更廣的 QR 符號。

---

## 生產環境的專業建議

- **盡早授權：** 評估版會加上小浮水印。於第一次呼叫 `BarCodeGenerator` 前載入授權檔，以免出現浮水印。  
- **批次處理：** 產生大量條碼時，可重複使用同一個 `ExtCodetextBuilder` 實例——每次迴圈只需呼叫 `clear()`。  
- **執行緒安全：** 每條執行緒應自行持有 `BarCodeGenerator` 實例，物件本身不具執行緒安全性。  
- **效能考量：** `x_dimension` 設得過低會增加渲染時間。若在高吞吐量環境，請以目標硬體進行基準測試。  
- **測試驗證：** 必須使用至少兩種不同讀取器（手機 App 與硬體掃描器）驗證條碼，確保 ECI 能被正確處理。
